# 飞书接入用户操作指南

这份指南给第一次使用工具的人看。目标是把 H5 小游戏评测结果自动写入飞书多维表格。

如果只是 POC 或临时测试，可以先不用接飞书 API，直接使用工具导出的 `feishu_payload_preview.json`、CSV 或 Excel。

## 一句话说明

自动写入飞书需要完成四件事：

1. 创建一个飞书自建应用。
2. 给应用开通多维表格权限并发布版本。
3. 把应用添加到目标多维表格的“文档应用”里，并设置为可编辑。
4. 在工具里填写 App ID、App Secret、表格 URL 或 app_token/table_id。

## 0. 什么时候不用做这些

如果你只想手动导入飞书，可以不用创建应用。

你可以让工具导出：

```text
feishu_payload_preview.json
CSV
Excel
```

然后手动复制或导入到飞书多维表格。

## 1. 创建飞书自建应用

进入：

```text
飞书开放平台 → 开发者后台 → 创建应用 → 企业自建应用
```

建议填写：

```text
应用名称：H5游戏评测助手
```

```text
应用描述：自动采集H5小游戏链接，生成评测字段、截图证据和飞书多维表格写入预览。
```

图标可以使用默认图标。建议选择紫色或蓝色背景。

注意：

- 不要写“官方”“飞书官方”“认证”等容易误导的词。
- 不要使用其他品牌 Logo。
- 个人飞书测试也可以创建自建应用。

## 2. 获取 App ID 和 App Secret

创建应用后，进入应用后台：

```text
左侧 → 凭证与基础信息
```

复制：

```text
App ID
App Secret
```

安全提醒：

- App ID 可以给开发人员。
- App Secret 不要发到聊天里。
- App Secret 只填写到本地配置文件或工具的模型/飞书配置页面。

## 3. 开通多维表格权限

进入：

```text
左侧 → 权限管理
```

搜索：

```text
多维表格
```

至少开通：

```text
查看、评论、编辑和管理多维表格
```

如果页面里有更细权限，优先选择包含“查看”和“编辑记录”的权限。

如果你的多维表格 URL 是 `/wiki/` 开头，还需要额外开通 Wiki 节点读取权限。最小权限建议：

```text
wiki:node:read
```

如果后台搜不到这个权限，可以开通任意一个等价权限：

```text
wiki:wiki:readonly
wiki:wiki
```

开通位置仍然在：

```text
左侧 → 权限管理
```

可以搜索：

```text
wiki
知识库
```

## 4. 发布应用版本

开通权限后，需要发布版本，否则配置可能不生效。

进入：

```text
左侧 → 版本管理与发布
```

点击：

```text
创建版本 → 发布
```

说明：

- 这是发布到你自己的飞书个人版/组织内部。
- 不是公开上架应用商店。
- 个人测试也需要发布版本让权限生效。

## 5. 把应用添加到多维表格

打开目标多维表格页面。

不要在“邀请协作者”的搜索框里搜应用，那里通常只能搜用户或群。

正确入口通常是：

```text
多维表格右上角 → ... 更多 → 添加文档应用 / 文档应用
```

搜索你创建的应用：

```text
H5游戏评测助手
```

添加后，权限设置为：

```text
可编辑
```

成功后会看到类似：

```text
已添加应用
H5游戏评测助手
可编辑
```

## 6. 获取多维表格 URL

打开多维表格，复制浏览器地址栏 URL。

普通多维表格 URL 常见格式：

```text
https://xxx.feishu.cn/base/bascnxxxxxxxx?table=tblxxxxxxxx
```

知识库里的多维表格 URL 常见格式：

```text
https://xxx.feishu.cn/wiki/<wiki_node_token>?table=<evaluation_table_id>
```

两者区别：

| URL 类型 | 是否能直接拿 app_token |
|---|---|
| `/base/...` | 可以直接从 `/base/` 后面拿 |
| `/wiki/...` | 不可以，需要工具调用飞书接口解析 |

## 7. 在工具里解析 URL

运行：

```bash
npm.cmd run poc:feishu:parse-url -- --url "你的多维表格URL"
```

输出：

```text
config/feishu_url_parse_result.json
```

如果 URL 是 `/base/`：

- 工具会直接解析出 `app_token` 和 `table_id`。

如果 URL 是 `/wiki/`：

- 工具会解析出 `wiki_node_token` 和 `table_id`。
- 还需要执行下一步，把 `wiki_node_token` 转成真正的 `app_token`。

## 8. 填写工具配置

复制模板：

```text
config/feishu.local.template.json
```

另存为：

```text
config/feishu.local.json
```

填写：

```json
{
  "enabled": true,
  "app_id": "你的 App ID",
  "app_secret": "你的 App Secret",
  "bitable": {
    "app_token": "base 链接可直接填；wiki 链接先留空，下一步解析",
    "tables": {
      "evaluation_results": {
        "table_id": "tbl..."
      }
    }
  },
  "write_mode": "preview"
}
```

## 9. 如果是 wiki 链接，解析 app_token

运行：

```bash
npm.cmd run poc:feishu:resolve-wiki
```

输出：

```text
config/feishu_wiki_resolve_result.json
```

成功后会得到：

```text
app_token
table_id
```

把 `app_token` 填回：

```text
config/feishu.local.json
```

当前 POC 工具会在解析成功后自动写回 `config/feishu.local.json`，正常情况下不需要手动复制。

如果解析失败，并且错误里出现：

```text
99991672
wiki:node:read
wiki:wiki:readonly
```

说明应用还没有开通 Wiki 节点读取权限。回到开放平台权限管理，开通 `wiki:node:read` 或 `wiki:wiki:readonly`，重新发布应用版本，再运行：

```bash
npm.cmd run poc:feishu:resolve-wiki
```

## 10. 检查配置

运行：

```bash
npm.cmd run poc:feishu:check
```

输出：

```text
config/feishu_config_check.json
```

常见状态：

| 状态 | 含义 |
|---|---|
| `template_ready` | 只检查了示例模板，还没填真实配置 |
| `ready_for_connection_test` | 本地配置形态正确，可以开始连飞书 |
| `invalid` | 必填项缺失或仍是占位符 |

## 11. 生成写入预览

运行：

```bash
npm.cmd run poc:feishu:preview
```

输出：

```text
evidence/<game_id>/feishu_payload_preview.json
```

这个文件就是后续要写入飞书的一条记录。

## 12. 可配置标签库

如果希望“玩法、题材、画风、特色标签、适合人群、操作方式”等选项在飞书里实时维护，可以使用统一标签库表。

先生成导入模板：

```bash
npm.cmd run poc:taxonomy:seed
```

输出：

```text
mock_bitable/taxonomy_options_seed.csv
```

在飞书多维表格里新建一个数据表，建议命名为：

```text
Taxonomy Options
```

把 `taxonomy_options_seed.csv` 导入进去。字段名保持不变：

```text
Category
Option ID
Parent ID
Level
Name EN
Name ZH
Enabled
Description ZH
Source File
```

然后把这个标签库表的 `table_id` 填到：

```text
config/feishu.local.json
bitable.tables.taxonomy_options.table_id
```

同步飞书标签库：

```bash
npm.cmd run poc:taxonomy:sync
```

也可以让工具自动创建 `Taxonomy Options` 表并导入初始选项：

```bash
npm.cmd run poc:taxonomy:setup-feishu -- --apply
```

同步成功后会生成：

```text
config/taxonomy_from_feishu.json
```

之后 AI 评测会优先使用飞书标签库；如果没有配置或同步失败，则继续使用本地 `mock_bitable/*.csv` 作为兜底。

## 13. 安全批量运行

单个游戏完整管线：

```bash
npm.cmd run poc:pipeline -- --game-id cow-saver --play-seconds 60 --ai local --trace off --write-feishu
```

说明：

- 已有采集证据时不会重复采集，除非加 `--force-collect`。
- 已有 AI 结果时不会重复评测，除非加 `--force-ai`。
- 写入飞书时按 `Game ID` / `Game URL` 查重，已有记录会更新，不会重复新增。
- 人工复核保存后，工具会尽量把 `Review Status`、`Review Notes`、`Review Updated At` 同步回同一条飞书记录；如果游戏还没有采集和 AI 结果，则只保存本地复核记录。
- 证据文件会自动保存到本地 `evidence/<game_id>/` 文件夹。截图默认只写本地路径；如果在配置页勾选“写入飞书时上传截图附件”，真实写入飞书时会同步到 `Screenshot Attachments` 附件字段。
- 视频证据只保存在本地 `video/` 子文件夹，不上传飞书。
- 默认 `--trace off`，避免部分 H5 游戏生成超大 trace 文件拖慢批量任务；需要调试问题游戏时再开启 `--trace minimal` 或 `--trace full`。
- 如果截图不完整，管线会标记 `success_with_review`，提醒先复核证据再用于正式评测。

批量计划：

```bash
npm.cmd run poc:batch:plan
```

批量 dry run：

```bash
npm.cmd run poc:batch:run -- --full-pipeline
```

批量实跑：

```bash
npm.cmd run poc:batch:run -- --full-pipeline --execute
```

## 14. 常见问题

### Gemini 连接超时怎么办

如果 `npm.cmd run poc:ai:test` 返回 `network_error`，并且错误码类似 `UND_ERR_CONNECT_TIMEOUT`，说明这台电脑当前不能直接访问 Gemini API。

可以在 `.env` 里增加代理：

```text
GEMINI_PROXY=http://127.0.0.1:7890
```

然后重新测试：

```bash
npm.cmd run poc:ai:test
```

为了控制消耗，正式批量前先用 1 张截图验证：

```bash
npm.cmd run poc:ai -- --game-id pixel-match --mode low --max-images 1
```

### 搜不到应用

不要在“添加协作者”里搜应用。请找：

```text
右上角 ... → 添加文档应用
```

### URL 是 wiki 开头怎么办

正常。知识库里的多维表格经常是 `/wiki/` 链接。工具会先解析出 `wiki_node_token`，再通过飞书接口换成真正的多维表格 `app_token`。

### 需要发布应用吗

需要。权限配置发布后才会生效。这个发布是内部发布，不是公开上架。

### App Secret 可以发给别人吗

不建议。只填到本地配置或工具后台里。

### 证据截图怎么给团队看

POC 阶段先保存本地路径。团队协作阶段需要把截图上传到飞书云文档、对象存储或内网文件服务，再把 URL 写入飞书表格。

## 15. 推荐配置顺序

```text
创建自建应用
→ 拿 App ID / App Secret
→ 开通多维表格权限
→ 如果 URL 是 wiki，额外开通 wiki:node:read
→ 发布版本
→ 添加文档应用到目标多维表格
→ 复制多维表格 URL
→ 运行 URL 解析
→ 填 feishu.local.json
→ 生成并导入 Taxonomy Options 标签库表（可选）
→ 同步飞书标签库（可选）
→ 运行配置检查
→ 生成写入预览
→ 执行真实写入
```
